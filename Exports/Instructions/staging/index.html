My App – Staging
Description

Staging environment of "My App" on a public cloud, integrated with cloud services

Overview

Our App is almost ready to be pushed to production and now is the time to make sure it works as expected in a staging environment. In this sandbox, we follow best practices for a production environment and use cloud services such as AWS Elastic Load Balancer and AWS CloudFront (CDN service). "My App" is deployed as a 2-tier application with multiple web servers for redundancy and a load balancer distributing the traffic between them.

Note that this blueprint is an offline template of the sandbox. In real life scenarios, such a blueprint is usually created by DevOps engineers to be used by testers, developers or by some CI process. It is even possible to integrate with pipeline tools that will push this staging sandbox, once approved, to production.

Sandbox components:

  • My App Web Server - the web server of the application "My App", installed on Linux Ubuntu
  • My SQL DB - the database of the application "My App", installed on Linux Ubuntu
  • AWS Elastic Load Balancer - a classic load balancer cloud service by AWS
  • AWS CloudFront - a global CDN cloud service by AWS, configured as the application front

Note that the component you see on the sandbox canvas are either "applications" (a.k.a "Apps") or "services". Those are terms used in CloudShell to describe the types of component within the sandbox, not to be confused with the "My App" sample application (which consists of multiple.App and service component in this sandbox).

You might have noticed the two different colored lines - a connection between two applications and a connection involving a cloud service are marked differently in CloudShell.

Let's start the Sandbox

Now it’s time to make the blueprint live and create your Staging sandbox by clicking the Reserve button. Launching the sandbox will deploy its components on new AWS instances located in a cloud subnet (in this case, an AWS VPC), which is dedicated to the sandbox and is completely isolated.

Please note that in your trial version you are limited to 2 hours per sandbox and up to 2 concurrent sandboxes.

What can I do in this Sandbox?
1 - Monitor

When the sandbox starts, CloudShell launches the sandbox’s Setup process. This process automatically deploys all the component in the sandbox in the selected public cloud, in this case AWS US East (N. Virginia) region, and then configures them to the baseline state. You can monitor the Setup progress by opening the Output console.

Note that the Setup process might take some time. Please do not end the sandbox before the Setup process completes.

2 - Access

Once the Setup has completed successfully, the sandbox will change its state to "Active". Now you can access the application. Since it is a staging environment, My App's database and web servers don't have a public IP and are only accessible via AWS CloudFront or via the Elastic Load Balancer.

To access My App's web interface, you can use the domain name link presented in the Output console after the Setup, or run the Get Access Information command on the "AWS CloudFront" service, which will return the domain name link in its output. It is also possible to access the "My App" application's web interface via the Elastic Load Balancer - just run the Get Access Information command on the "AWS Elastic Load Balancer" service and use the domain name link in the command's output.

As this is a staging environment, the database and web servers themselves are accessible only in SSH via a proxy called QualiX, which has access to the sandbox's cloud subnet. To access them via SSH, hover over the appropriate App and click the SSH link.

3 - Interact

Each application comes with some default commands (that can be extended in the full version of CloudShell).
The commands can be found in the Commands side pane. To open the side pane, hover over the application and click the Commands button.

You may run any of the default commands.

This sandbox also includes a custom sandbox command called Perform Healthcheck, which verifies that the web application is available both via the ELB and via the CDN (CloudFront), and updates their status accordingly. You can try it out, power off the MySQL DB VM and run the health-check command.

Note that the public IP might change when you power the VM off and then on again. This will cause the access links to break and can be fixed by running the "Refresh IP" command.

4 - Teardown

By default, the sandbox will automatically end once its allocated time has passed. However, you can manually stop the sandbox when you finish working with it by clicking the End button.
When the sandbox ends, a Teardown process kicks in and deletes all the VMs along with the sandbox's dedicated VPC and Subnet in AWS. You can monitor the Teardown progress by opening the Output console.
