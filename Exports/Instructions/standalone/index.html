My App – Standalone
Description

A pre-production standalone deployment of "My App" on a public cloud infrastructure.

Overview

Hi there! Welcome to your very first sandbox. In this sandbox, you’ll find a standalone version of your application “My App”. "My App" is a PHP-based eCommerce platform.

In this simple sandbox, both the web server and the database are installed on the same machine as a LAMP stack (Linux, Apache, MySQL, PHP) using an image of a LAMP stack VM from AWS marketplace.

A sandbox is based on a blueprint, which is an offline template of the sandbox, including the required components, connectivity and automation. In real life scenarios, blueprints are usually created by DevOps engineers to be used by testers, developers or sales engineers.

The blueprint can be deployed automatically by some CI process or deployed manually by a tester that wants to get a specific build of the application up and running without all the hassle of configuring everything manually.

Let's start the Sandbox

Now it’s time to make the blueprint live and create your first sandbox by clicking the Reserve button. Launching the sandbox will deploy and configure "My App" on a new AWS instance located in a cloud subnet (in this case, an AWS VPC), which is dedicated to the sandbox and is completely isolated.

Please note that in your trial version you are limited to 2 hours per sandbox and up to 2 concurrent sandboxes.

What can I do in this Sandbox?
1 - Monitor

When the sandbox starts, CloudShell automatically calls the sandbox’s Setup process. This process automatically deploys the VM in the selected public cloud in this case, AWS US East (N. Virginia) region, and then configures the application to the baseline state. You can monitor the Setup progress by opening the Output console.

Note that the Setup process might take some time. Please do not end the sandbox before the Setup process completes.

2 - Access

Once the Setup has completed successfully, the sandbox will change its state to "Active" (take a look at the main menu's blue bar). You can access the VM by hovering over the App and clicking the SSH link, or open the web interface of the application by clicking Web.

Note that the IP address you see on the deployed application is the private IP inside the sandbox's subnet. The application cannot be accessed using this IP address, however, your App also has a public IP, which can be found in the App's Attributes side pane with specific ports that allow access to the App's web interface. To open the side pane, hover over the App and click the Attributes button. The Web link above utilizes the public IP address and port.

To access the App's administrator page, hover over the App and click the Web Management link. Log in with the following credentials:
User: trial@quali.com
Password: MyApp@Cloudshell

3 - Interact

The application comes with some default commands, which can be extended in the full version of CloudShell. The commands can be found in the Commands side pane. To open the side pane, hover over the App and click the Commands button.

Note that the public IP might change when you power the VM off and then on again. This will cause the access links to break and can be fixed by running the "Refresh IP" command..

You may run any of the default commands.

4 - Teardown

By default, the sandbox is set to automatically end once its allocated time has passed. However, you can manually stop the sandbox when you finish working with it by clicking the End button.

When the sandbox ends, a Teardown process kicks in and deletes all the VMs along with the sandbox's dedicated VPC and Subnet on AWS. You can monitor the Teardown progress by opening the Output console.
