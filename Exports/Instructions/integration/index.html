My App - Integration
Description

Testing blueprint for a 2-tier deployment of "My App" on a public cloud infrastructure.

Overview

It is time to move to the next stage in our application development flow. In this sandbox, we will test the application using an external UI testing tool called Selenium. The App will be deployed as a 2-tier application, meaning that the web server and database are installed separately on two different VMs. No worries, all the configuration magic will happen automatically during the Setup phase of the sandbox.
The sandbox includes the following components:

  • My App Web Server - the web server of the application "My App", installed on Linux Ubuntu
  • My SQL DB - the database of the application "My App", installed on Linux Ubuntu
  • Selenium Hub - a Selenium hub installed on Amazon Linux AMI. Selenium is a UI testing tool. The hub receives the test request and distribute it to the right node.
  • Selenium Node - a Selenium node installed on a Windows VM. The node executes the test.

Note that the components you see on the sandbox canvas are called "Applications" or "Apps" in CloudShell terminology, not to be confused with the "My App" sample application (which consists of multiple App components in this sandbox).

The Setup process of this sandbox configures the Selenium hub to take the tests from an external test repository in GitHub. Once the Setup process completes, it will be possible to launch a Selenium test via a command available in the sandbox, and the test result will be returned to CloudShell and available for the end user. This usage of Selenium is just an example of how you can easily customize the CloudShell platform and integrate with any tool or service.

Let's start the Sandbox

Now it’s time to make the blueprint live and start the sandbox by clicking the Reserve button. Launching the sandbox will deploy its components on new AWS instances located in a cloud subnet (in this case an AWS VPC), which is dedicated to the sandbox and completely isolated.

Please note that in your trial version you are limited to 2 hours per sandbox and up to 2 concurrent sandboxes.

What can I do in this Sandbox?
1 - Monitor

When the sandbox starts, CloudShell launches the sandbox’s Setup process. This process automatically deploys all the components in the sandbox into the selected public cloud, in this case AWS US East (N. Virginia) region, and then configures them to the baseline state. You can monitor the Setup progress by opening the Output console.

Note that the Setup process might take some time. Please do not end the sandbox before the Setup process completes.

2 - Access

Once the Setup has completed successfully, the sandbox will change its state to "Active". Now you can access the applications:

  • Open the web interface of the "My App" application by clicking the App's Web link

3 - Run Test

This integration sandbox includes a "Run Test" command that finds both the Selenium Hub and the application, and launches a predefined Selenium test. The test is launched on the Hub that executes the tests in an available Selenium Node.

The command can be found in the Commands side pane on the right. To open the side pane, click the Commands button in the top menu.

The Selenium Hub in this example is configured to work with a GitHub repository containing two tests - one called "Check_Homepage", which verifies that the web interface is up and takes a screenshot, the other called "Find_Text_In_Homepage", which looks for the string "Latest Products" in the homepage. To run a command, just click the Play button, fill in the test name and click Run.

Once the test ends, you will get a message in the Output console and the test result will be attached to the sandbox (refresh the page to see the attachment link).

You can try and interfere with the test by powering off the web server or database and rerunning the test to make sure it fails in such scenarios. To run the test, open the Commands side pane, fill in the test name and launch the Run Test command.

4 - Teardown

By default, the sandbox is set to automatically end once its allocated time has passed. However, you can manually stop the sandbox when you finish working with it by clicking the End button.

When the sandbox ends, a Teardown process kicks in and deletes all the VMs along with the sandbox's dedicated VPC and Subnet in AWS. You can monitor the Teardown progress by opening the Output console.

Example flow
  1. Reserve the Blueprint
  2. Open the Output console to monitor the Setup progress
  3. Wait for the Setup process to finish and for the sandbox to become Active
  4. Open the Web interface of "My App" and view the application
  5. Run the "Check_Homepage" test using the Run Test command
  6. See that the test completes successfully and read the attached report
  7. Run the Power off command on the "My App Web Server" application, rerun the test and see that it fails
  8. End the sandbox and monitor the Teardown progress using the Output Console
