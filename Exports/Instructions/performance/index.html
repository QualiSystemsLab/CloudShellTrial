My App - Performance
Description

Blueprint that simulates and tests HA configuration of "My App", deployed on public cloud infrastructure

Overview

We're almost there! Our App will soon get to the staging environment for final touches before production. But first thing first, we want to make sure our App is robust and configured with the right redundancy and load balancing. In this sandbox, you’ll find a 2-tier version of your application “My App” - a database and a web server; but this time, we deploy several identical instances of the web server for redundancy and use a load balancer called HA Proxy to distribute the traffic between them.

Note that this blueprint is an offline template of the sandbox. In real life scenarios, such a blueprint is usually created by DevOps engineers to be used by testers or developers. The blueprint can be deployed automatically by some CI process or deployed manually by a tester or developer that wants to get a specific build of the application installed as a 2-tier with HA without having to manually and repeatedly go through the pain of setting up such an environment.

The sandbox includes the following components:

  • My App Web Server - the web server of the application "My App", installed on Linux Ubuntu
  • My SQL DB - the database of the application "My App", installed on Linux Ubuntu
  • HA Proxy - a load balancer and proxy software installed on Amazon Linux AMI
  • JMeter - a load testing tool installed on a Windows VM

Note that the component you see on the sandbox canvas are called "applications" or "Apps" in CloudShell terminology, not to be confused with the "My App" sample application (which consists of multiple App component in this sandbox).

Let's start the Sandbox

Now it’s time to make the blueprint live and create your sandbox by clicking the Reserve button. Launching the sandbox will deploy its components on new AWS instances located in a cloud subnet (in this case, in an AWS VPC) which is dedicated to the sandbox and is completely isolated.

Please note that in your trial version you are limited to 2 hours per sandbox and up to 2 concurrent sandboxes.

What can I do in this Sandbox?
1 - Monitor

When the sandbox starts, CloudShell launches the sandbox’s Setup process. This process automatically deploys all the components in the sandbox in the selected public cloud, in this case AWS US East (N. Virginia) region, and then configures them to the baseline state. You can monitor the Setup progress by opening the Output console.

Note that the Setup process might take some time. Please do not end the sandbox before the Setup process completes.

2 - Access and Validate

Once the Setup has completed successfully, the sandbox will change its state to "Active". Now you can access the applications - browse the web interface of the "My App" application via each one of the web servers or via the "HA Proxy", open MySQL's PHPMyAdmin interface by clicking its Web link

Note that in this sandbox, all the components got a public IP for debugging purposes. Obviously that won't be the situation in a production environment in which only the proxy will get a public IP and that is indeed the configuration in the Staging sandbox.

3 - Run a performance test

This performance sandbox includes a "Run JMeter Test" command that uses the JMeter component to test My App via the HA Proxy load balancer.
This is performed by launching a predefined JMeter test. 

The command can be found in the Commands side pane on the right. To open the side pane, click the Commands button in the top menu.

The JMeter component in this example is configured to work with a GitHub repository containing a test called "find_mug", which simulates a user flow that opens the "Mugs" page within My App and searches for a mug named "quia". To run the test just click the Play button on the command, fill in the test name in the "Test_Name" input and the amount of concurrent users you want to simulate in the "Thread_Count" input (expects a number, eg. 10) and then click Run.

Once the test is completed, you will get a message in the Output console and the test result will be attached to the sandbox (refresh the page to see the attachment link).

If you are familiar with JMeter feel free to open the JMeter UI directly.
Click the RDP link on the JMeter component to connect to the AWS Instance running JMeter, and open the JMeter application via the shortcut on the desktop.
Within JMeter UI you can write and launch your own load test or use the predefined test that we created for this sandbox, which can be found in our CloudShell VE Trial GitHub repo at https://github.com/QualiSystemsLab/CloudShellTrial/tree/master/JMeterTests

4 - Interact

Each application comes with some default commands that can be extended in the full version of CloudShell.
The commands can be found in the Commands side pane. To open the side pane, hover over the application and click the Commands button.

You may run any of the default commands.

Note that the public IP might change when you power the VM off and then on again. This will cause the access links to break and can be fixed by running the "Refresh IP" command.

5 - Teardown

By default, the sandbox is set to automatically end once its allocated time has passed. However, you can manually stop the sandbox when you finish working with it by clicking the End button.

When the sandbox ends, a Teardown process kicks in and deletes all the VMs along with the sandbox's dedicated VPC and Subnet in AWS. You can monitor the Teardown progress by opening the Output console.

Example flow
  1. Reserve the Blueprint
  2. Open the Output console to monitor the Setup progress
  3. Wait for the Setup process to finish and for the sandbox to become Active
  4. Open the Web interface of "My App" via the "Web" button on the "HA Proxy"
  5. Find the Web Server you got redirected to according to web platform's background color.
  6. End the reservation and monitor the Teardown process using the Output Console
